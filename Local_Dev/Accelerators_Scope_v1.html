Design Principles

Each principle is derived by inverting a known failure mode in accelerator initiatives

▲ Accelerators become a "platform" nobody asked for
Composable Building Blocks, Not a Platform
Each accelerator is independently adoptable and ejectable. No monolith.
▲ Over-abstraction makes nothing actually useful
Useful on Day One
Every component must solve a real, immediate problem on the current engagement.
▲ Hard-coded assumptions don't transfer
Configuration Over Code
Behavior driven by YAML/JSON configs, not embedded logic.
▲ Teams skip testing because "it's just a chatbot"
If You Can't Measure It, Don't Ship It
Evaluation is a first-class concern, not an afterthought.
▲ Accelerators rot because nobody maintains them
Built Through Real Projects
Extracted from working implementations, not theorized in isolation.
▲ Scope creep turns 2 weeks into 2 months
Start Narrow, Widen With Evidence
Begin with the first use case; generalize only when a second demands it.

Detailed Accelerator Scope

Deep-dive into the first two accelerators being built

Data Layer Accelerator

Compresses client data understanding from weeks to days — connectors, profiling, quality assessment, and schema discovery in one configurable toolkit.

Core Components

ComponentWhat It DoesWhy It Matters
Data Connector LibraryPre-built, config-driven connectors for common databases (PostgreSQL, MySQL, SQL Server, Snowflake, BigQuery, MongoDB) and file formats (CSV, Excel, Parquet, JSON). Standardized output regardless of source.Eliminates custom ingestion code. Connect via a YAML config, not custom code.
Schema Discovery EngineAuto-introspects database to discover tables, columns, types, primary keys, foreign keys, indexes, and inferred relationships (even when FKs are not explicitly defined).Structural understanding of a client's data model in minutes, not days.
Data Profiling EngineStatistical profiling: cardinality, null rates, value distributions, outlier detection, pattern recognition (e.g., "this column looks like an email/phone/date").Schema shows structure; profiling shows reality. An agent unaware of 40% NULLs will generate misleading queries.
Data Quality AssessorScores data across four dimensions — completeness, consistency, accuracy, timeliness — with specific findings and remediation guidance.Sets expectations early. Know data quality before building the agent, not after deploying it.

Inputs & Outputs

Inputs
  • Database connection config (YAML/JSON)
  • OR file paths (CSV, Excel, Parquet, JSON)
→
Outputs
  • Schema map (tables, columns, relationships)
  • Data profiling report (per-column statistics)
  • Quality scorecard (4-dimension scoring)
  • JSON (machine) + Markdown (human) formats

Scope Boundaries

In Scope
  • Connecting to client data sources
  • Schema extraction and relationship inference
  • Statistical profiling and quality scoring
  • Quality reports with remediation guidance
  • Support for structured data (databases, files)
Out of Scope
  • Data migration or ETL pipelines
  • Data transformation or cleansing
  • Real-time streaming data ingestion
  • Building or managing a data warehouse
  • Unstructured document parsing (separate layer)

Technical Approach

  • Config-drivenAll connections defined in YAML/JSON — no code changes to onboard a new data source
  • Provider-agnosticSQLAlchemy-based for relational databases; dedicated adapters for cloud warehouses where needed
  • Profiling engineSQL-based sampling for large datasets; in-memory analysis for smaller files
  • Quality frameworkPluggable rules with sensible defaults; engagement-specific rules can be added
  • Output formatDual output — JSON for downstream accelerators; markdown for human review

What Can Go Wrong — Inversion Analysis

Risk
Connector sprawl — supporting every database becomes a maintenance burden
Mitigation
Start with the 4–5 most common. Add new connectors only when an engagement demands it.
Risk
False confidence from profiling — automated profiles may miss semantic issues
Mitigation
Reports include "human review needed" flags. Profiling informs; it does not replace domain understanding.
Risk
Performance on large databases — profiling 500+ tables at scale
Mitigation
Sampling-based profiling by default. Configurable sample sizes and table subsets.
Risk
Quality scores without guidance — numbers without actionable insight
Mitigation
Every finding includes: what is wrong, why it matters for agents, and what to do about it.
Risk
Overfitting to SQL databases — not every client has structured data
Mitigation
Deliberately focused on structured data. Unstructured handling is a separate layer.

Success Criteria

  • A new client database can be connected and fully profiled in under 2 hours
  • Schema map is accurate enough to feed directly into the Knowledge Graph Assembler
  • Quality scorecard surfaces at least 80% of data issues that would affect agent performance
  • Output formats (JSON + markdown) are stable and consumed by downstream accelerators

Agent Framework Accelerator

Reusable building blocks for assembling single and multi-agent systems — so the team focuses on business logic, not plumbing.

Core Components

ComponentWhat It DoesWhy It Matters
Base Agent TemplatesPre-wired templates for NL2SQL, Conversational QA, Plan-and-Execute, ReAct, and Workflow agents. Each includes tool binding, output parsing, error handling, and logging.Starting from a working template is 10x faster than building from scratch.
Agent OrchestratorMulti-agent coordination: Sequential, Parallel, Hierarchical (supervisor), Router-based. State management and handoff protocols included.Proven patterns prevent ad-hoc, fragile wiring in multi-agent systems.
Prompt ManagementCentralized registry with versioning, Jinja2 templating, few-shot banks, system/user/tool separation. File-based, no external dependencies.Prompts are the "code" of AI apps. Scattered, un-versioned prompts are the #1 maintenance headache.
Tool & Function RegistryPre-built tools: SQL executor (with safety), chart generator, REST caller, sandboxed code executor, file reader/writer. Standardized interface.Pre-built + safety checks eliminate repeated work and reduce risk.
LLM GatewayAbstraction over providers (OpenAI, Anthropic, Google, Azure, Ollama). Unified API, fallback chains, rate limiting, token tracking.Provider changes become a config switch, not a code rewrite.
Memory & ContextPluggable memory: buffer, summary, sliding window, entity extraction. Backends: in-memory, Redis, PostgreSQL. Auto context window management.The coherence layer — prevents context overflow and irrelevant history.

Inputs & Outputs

Inputs
  • Agent config (YAML/JSON): type, model, tools, memory, prompts
  • Orchestration graph config (for multi-agent setups)
  • Prompt templates + few-shot examples
→
Outputs
  • Runnable agent instance (tools, prompts, memory wired)
  • Multi-agent system with managed execution
  • Versioned prompt registry

Scope Boundaries

In Scope
  • Agent templates for common patterns
  • Multi-agent orchestration patterns
  • Centralized prompt management with versioning
  • Pre-built tools with standard interfaces
  • LLM provider abstraction and routing
  • Pluggable memory backends
  • Configuration-driven agent assembly
Out of Scope
  • Fully custom agent architectures
  • Distributed agents across networks
  • Automated prompt optimization
  • Domain-specific business logic tools
  • Model fine-tuning or training
  • Custom memory compression models
  • GUI-based agent builder (future)

Technical Approach

  • Config-driven assemblyAgents defined in YAML/JSON. Framework assembles runnable instances. No framework lock-in.
  • Framework-agnosticLangGraph as primary engine, but config layer abstracts it. Swapping engines changes only the adapter.
  • Prompt-as-dataPrompts in structured file hierarchy, git-tracked, Jinja2-templated. Few-shots loaded per domain.
  • Tool contractStandard interface: name, description, input_schema, output_schema, execute. Discoverable and composable.
  • Gradual adoptionEach component works independently. Use any piece without the others.

What Can Go Wrong — Inversion Analysis

Risk
Framework bottleneck — abstraction adds latency or limits capability
Mitigation
Thin abstraction. Drop to raw LangGraph or direct API anytime. The framework helps; it never blocks.
Risk
Template rigidity — every engagement forks templates
Mitigation
Inheritance model: base skeleton + config overrides. If >30% changes, create a new template type.
Risk
Prompt management overhead — centralization adds process
Mitigation
File-based, no database or service. Works with existing git workflow. Near-zero overhead.
Risk
Tool safety gaps — SQL executor runs destructive queries
Mitigation
SELECT-only allowlist by default. Destructive ops need explicit config opt-in. Code executor isolated.
Risk
Gateway hides provider features
Mitigation
Common API for common ops. Provider-specific features accessible via pass-through.
Risk
Over-engineering for single-agent cases
Mitigation
Orchestrator is optional. Single agent? Use a template directly.

Success Criteria

  • A new NL2SQL agent assembled from config + prompts in under a day
  • Switching the LLM provider is a config change, not a code change
  • Adding a new tool means adding a definition file, not modifying agent internals
  • Prompts are version-controlled, templated, and shared without duplication
  • At least two agent types built from the same framework, proving reusability
